# Component Props

Props are a mechanism for passing data from a parent component to a child component.

They allow components to be configured and reused while maintaining a predictable one-way flow of data throughout your application.

---

## Declaring Props

Props are accessed through the `props` parameter:

```js
export default function(props) {}
```

The `props` object contains all values passed to the component.

### Parent Component

```html
<UserCard name="John" />
```

### Child Component

```js
export default function(props) {
    console.log(props.name); // John
}
```

---

## Static Props

Static props pass literal values to a component.

### Parent

```html
<UserCard name="John" age="25" />
```

### Child

```js
export default function(props) {
    console.log(props.name);
    console.log(props.age);
}
```

Static props are evaluated once when the component is created.

---

## Reactive Props

Props can be bound to reactive values using the `:` prefix.

### Parent

```js
const [name, setName] = signal("John");
```

```html
<UserCard :name="name()" />
```

When the signal changes:

```js
setName("Jane");
```

the child automatically receives the updated value.

```text
Parent State
      ↓
     Prop
      ↓
Child Component
```

---

## One-Way Data Flow

Props always flow from parent to child.

```text
Parent
    ↓
 Child
```

When the parent updates:

```js
setName("Jane");
```

the child receives the new value.

However, the child should not directly modify props.

```js
props.name = "Bob";
```

Attempting to mutate a prop throws an error because props are read-only.

Instead, the parent remains the source of truth.

If a child needs to communicate changes back to a parent, consider using:

* Events
* Signals
* Context
* Shared state

depending on the use case.

---

## Example

The child component communicates changes back to the parent by receiving both the signal getter and setter.

### Parent

```js
const [count, setCount] = signal(0);
```

```html
<Counter :count="count" :setCount="setCount" />
```

### Child

```html
<button
    on:click="() => props.setCount(props.count() + 1)"
>
    Count is: {{ props.count() }}
</button>
```

The child does not own the state.

Instead, it receives the state through props and updates it using functions provided by the parent.

---

## Props Are Read-Only

All props are immutable inside the child component.

Core enforces this to ensure predictable data flow and avoid unintended side effects.

If mutation is required, it should be performed by the parent or through explicitly provided APIs.

---

## Passing Signals

Signals can be passed directly as props.

### Parent

```js
const [count, setCount] = signal(0);
```

```html
<Counter :count="count" :setCount="setCount" />
```

### Child

```js
export default function(props) {
    const { count, setCount } = props;
}
```

```html
<button on:click="() => setCount(count() + 1)">
    Count: {{ count() }}
</button>
```

Because `count` is a signal getter and `setCount` is a signal setter, destructuring is safe.

> [!NOTE]
> Reactivity is established when `count()` is executed, not when the function reference is destructured.

---

## Passing Proxy Objects

Reactive objects and arrays may also be passed as props.

### Parent

```js
const [user] = signal({
    name: "John",
    age: 25
});
```

```html
<UserCard :user="user()" />
```

Executing:

```js
user()
```

returns a reactive proxy object.

### Child

```js
export default function(props) {
    const { user } = props;
}
```

```html
<h1>{{ user.name }}</h1>
<p>{{ user.age }}</p>
```

Updates to the proxy remain reactive.

```js
user.name = "Jane";
```

automatically updates any reactive consumers.

Because the proxy itself is reactive, destructuring the prop does not break reactivity.

---

## Safe Destructuring

The following values can safely be destructured:

### Signal Getters

```js
const { count } = props;

count();
```

### Signal Setters

```js
const { setCount } = props;

setCount(1);
```

### Proxy Objects

```js
const { user } = props;

console.log(user.name);
```

These remain reactive because the reactive value itself is being passed.

---

## Unsafe Destruction

Destructuring can break reactivity depending on what is being passed.

### ❌ Non-Reactive Access

Parent:

```html
<script>
    import { signal } from "core";

    export default function() {
        const [name, setName] = signal("John");
    }
</script>

<Card :name="name()" />
````

Child:

```js
export default function(props) {
    const { name } = props;
}
```

```html
<h1>{{ name }}</h1>
```

In this example, the parent executes the signal getter:

```js
name()
```

and passes the resulting value to the child.

The child therefore receives:

```js
props.name === "John"
```

not:

```js
props.name === name
```

When destructured:

```js
const { name } = props;
```

the variable becomes a local reference to the current value.

Future updates to the prop will update:

```js
props.name
```

but will not update:

```js
name
```

As a result, reactivity is lost.

---

### ✅ Reactive Access

```js
export default function(props) {}
```

```html
<h1>{{ props.name }}</h1>
```

Accessing the value through the `props` object allows Core to track prop updates correctly.

When the parent updates:

```js
setName("Jane");
```

the child automatically receives the new value.

---

### Why Signal Getters Work Differently

The following remains reactive:

```html
<Card :name="name" />
```

because the signal getter itself is being passed.

```js
props.name === name
```

The child can safely destructure:

```js
const { name } = props;
```

and later execute:

```js
name()
```

to access the current value.

In this case, reactivity comes from the signal getter rather than the `props` object.

---

### Why Proxy Objects Work Differently

The following also remains reactive:

```html
<UserCard :user="user()" />
```

when `user()` getter is executed it returns a reactive proxy object.

```js
const { user } = props;
```

is safe because the proxy itself remains reactive regardless of how it is referenced.

```html
<h1>{{ user.name }}</h1>
```

will continue to update when the proxy changes.

---

## Rule of Thumb

The following are generally safe to destructure:

```js
const { count } = props;     // signal getter
const { setCount } = props;  // signal setter
const { user } = props;      // proxy object
```

The following should generally remain on the `props` object:

```js
props.name
props.age
props.title
```

unless you know the value being passed is itself reactive.

When in doubt, access values through `props`.

---

## Default Values

Core currently does not provide a built-in mechanism for defining default prop values.

If a default value is needed, it can be handled manually:

```js
export default function(props) {
    const title = props.title ?? "Untitled";
}
```
