# Control Flow

Core provides built-in control flow syntax that allows you to conditionally render parts of the DOM based on reactive state. The primary control flow directives are:

- `{{#if}}` — conditional rendering
- `{{#each}}` — list rendering
- `{{#await}}` — asynchronous rendering

These directives are part of Core's template system and are compiled into optimized runtime instructions.

---

## `{{#if}}` Conditional Rendering

The `{{#if}}` directive is used to conditionally render a section of the template based on a reactive expression.

```html
{{#if show}}
    <h1>Hello Core</h1>
{{/if}}
```

You can also provide an `else` branch:

```html
{{#if show}}
    <h1>Visible</h1>
{{:else}}
    <h1>Hidden</h1>
{{/if}}
```

You can also provide an `else-if` branch:

```html
{{#if show}}
    <h1>Visible</h1>
{{:else if show_alternate}}
    <h1>Alternate</h1>
{{/if}}
```

### How It Works

When the condition changes, Core evaluates the block and efficiently mounts or unmounts the corresponding DOM segment.

```
state change → evaluate condition → mount/unmount block
```

Only the affected section of the DOM is updated.

---

## `{{#each}}` List Rendering

The `{{#each}}` directive is used to render lists of items.

```html
{{#each items as item}}
    <p>{{ item().name }}</p>
{{/each}}
```

You can also access the index:

```html
{{#each items as item, index}}
    <p>{{ index }} - {{ item().name }}</p>
{{/each}}
```

### Empty State Handling

Core provides a `{{:empty}}` directive to handle cases where the list has no items.

```html
{{#each items as item, index}}
    <p>{{ index }} - {{ item().name }}</p>
{{:empty}}
    <p>Nothing to show</p>
{{/each}}
```

> [!NOTE]
> In list rendering, each element in a collection is treated as a **value accessor function**, not a direct variable.
>
> This is done so Core can intercept reads in the same way signals are tracked, enabling fine-grained reactivity even inside loops.
>
> This design is intentional—even when `items` is not reactive because it keeps list rendering behavior consistent with the rest of Core's reactivity system.

### How It Works

Core tracks each item and updates only the changed nodes when the array is mutated.

```
array mutation → update affected nodes → patch DOM
```

---


## `{{#await}}` — Async Rendering

The `{{#await}}` directive is used to handle asynchronous values such as Promises or async data sources.

```html
{{#await user}}
    <p>Loading...</p>
{{:then user}}
    <p>Hello {{ user.name }}</p>
{{/then}}
```

### `{{:then}}` and `{{:catch}}`

The `{{:then}}` and `{{:catch}}` directive handle resolved and rejected states respectively.

```html
{{#await data}}
    <p>Loading...</p>
{{:then result}}
    <p>Success: {{ result }}</p>
{{:catch error}}
    <p>Error: {{ error.message }}</p>
{{/await}}
```

### Optional Bindings

Both `{{:then}}` and `{{:catch}}` may be used without explicitly naming the value:

```html
{{#await data}}
    <p>Loading...</p>
{{:then}}
    <p>Loaded successfully</p>
{{:catch}}
    <p>Something went wrong</p>
{{/await}}
```

### How It Works

Core automatically tracks the latest promise state transitions:

```
pending → fulfilled → rejected
```

Each state renders the corresponding template block.

---

## Combining Control Flow

Control flow directives can be nested and combined:

```
{{#if users}}
    {{#each users}}
        {{#if active}}
            <p>{{ name }}</p>
        {{/if}}
    {{:empty}}
        <p>is empty</p>
    {{/each}}
{{/if}}

{{#await userPromise}}
    <p>Loading...</p>
{{#then user}}
    <h1>{{ user.name }}</h1>
{{/await}}
```

Core compiles nested blocks into an optimized execution structure to minimize unnecessary DOM operations.

---

## Key Differences from Traditional Rendering

Unlike virtual DOM frameworks, Core does not re-render entire components when state changes.

Instead:

- Each block is compiled into a render function
- Only dependent DOM regions are updated
- Control flow is resolved at a granular level

---

## Pluggable Control Flow Syntax

Core separates **reactivity** from **control flow syntax**.

This means control flow is not hard-coded into the framework. Instead, it is implemented as a template-layer feature that can be extended, replaced, or customized.

By default, Core provides a Handlebars-style syntax:

```html
{{#if condition}}
    <p>Hello</p>
{{/if}}

{{#each items as item}}
    <p>{{ item() }}</p>
{{/each}}
```

However, the control flow system is intentionally **decoupled from the runtime reactivity system**, allowing alternative syntaxes to be introduced without changing Core’s execution model.

### Alternative Syntax Support

Core can support multiple directive styles depending on the template compiler configuration, such as:

```html
<!-- Alpine-style -->
<div x-if="condition"></div>
<div x-for="item in items"></div>

<!-- Blazor-style -->
<div @if="condition"></div>
<div @for="item of items"></div>
```

These syntaxes are not special-cased in the runtime.

Instead, they are translated by the template compiler into a shared internal representation.

---

### Why This Design Matters

By keeping control flow independent from the runtime, Core gains several benefits:

- The runtime remains small and stable
- New syntaxes can be introduced without changing the reactivity engine
- Templates can be compiled into a unified instruction format regardless of syntax style
- Framework behavior stays consistent across different template dialects

---

### Internal Representation

Regardless of syntax, all control flow constructs are compiled into the same internal block structure:

```
IF_BLOCK
EACH_BLOCK
AWAIT_BLOCK
```

The runtime only understands this normalized representation—not the original syntax.

---

### Extensibility

Because of this separation, developers can theoretically:

- Implement custom template syntaxes
- Build domain-specific templating layers
- Port familiar syntax from other frameworks
- Create opinionated DSLs on top of Core

without modifying the runtime system.
