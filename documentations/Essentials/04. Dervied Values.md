# Derived Values

In Core, not all state needs to be stored directly. Often, values can be derived from existing signals. These are called **derived values**.

Derived values allow you to compute state on-demand while keeping it in sync with reactive dependencies automatically.

## Basic Derived Values

The simplest form of a derived value is a function that reads from one or more signals:

```js
const [count, setCount] = signal(16);

const double = () => count() * 2;
````

Whenever `count` changes, any reactive context using `double()` will automatically update.

```js 
console.log(double());      // 32
```

This works because derived functions are tracked like any other reactive computation.

---

## Why Derived Values Work

When `count()` is called inside a function:

```js
const double = () => count() * 2;
```

Core registers a dependency between:

```
double → count
```

This means:

* `double()` is re-evaluated when `count` changes
* Any effect using `double()` is also updated

---

## Memoized Derived Values

While function-based derived values are simple, they are recomputed every time they are accessed.

For expensive computations, Core provides the `memo()` function to cache results and avoid unnecessary recalculation.

```js
const [count] = signal(0);

const double = memo(() => count() * 2);
```

Unlike a plain function, a memoized value:

* Tracks dependencies
* Caches the computed result
* Recomputes only when dependencies change

---

## Why Use memo?

Consider a heavy computation:

```js
const expensive = memo(() => {
    let result = 0;

    for (let i = 0; i < 1e6; i++) {
        result += count();
    }

    return result;
});
```

Without `memo()`, this calculation would run every time it is accessed.

With `memo()`:

* The computation runs once initially
* Re-runs only when `count()` changes
* Cached value is reused across reads

---

## Difference: Derived Function vs Memo

### Derived Function

```js
const value = () => count() * 2;
```

* Recomputed every access
* No caching
* Lightweight

---

### Memoized Value

```js
const value = memo(() => count() * 2);
```

* Cached result
* Recomputes only on dependency change
* Optimized for expensive computations

---

## When to Use Each

Use a **derived function** when:

* Computation is cheap
* You want simplicity
* Value is used in a small scope

Use `memo()` when:

* Computation is expensive
* Value is accessed frequently
* You want to avoid repeated recalculation

--- 

## Asynchronous Updates and Batching

Unlike plain derived functions, memoized values are updated asynchronously.

When a dependency changes, Core does not immediately re-run the memo computation. Instead, the update is scheduled and batched together with other reactive updates.

```js
const [count, setCount] = signal(0);

const double = memo(() => {
    console.log("recomputed");
    return count() * 2;
});

setCount(1);
setCount(2);
setCount(3);
```

Even though `count` changed three times, the memo only recomputes once using the latest value.

```text
recomputed
```

This batching behavior helps avoid unnecessary work when multiple updates occur during the same execution cycle.

---

## Why Updates Are Deferred

Consider an expensive computation:

```js
const total = memo(() => {
    return calculateLargeDataset(items());
});
```

If the memo recomputed immediately for every dependency update:

```js
setItems(a);
setItems(b);
setItems(c);
```

the computation could run three separate times.

Instead, Core batches the updates:

```text
items → a
items → b
items → c
       ↓
single memo recomputation
```

allowing the memo to run only once using the final state.

---

## Memo Updates Before UI Updates

Core processes memoized values before updating the user interface.

The update order is:

```text
Signal Changes
       ↓
Memo Updates
       ↓
Effects
       ↓
DOM Updates
```

This guarantees that any effect or template expression reading a memoized value always receives the latest computed result.

For example:

```js
const [count, setCount] = signal(0);

const double = memo(() => count() * 2);

effect(() => {
    console.log(double());
});

setCount(5);
```

The memo updates first:

```text
count = 5
       ↓
double = 10
       ↓
effect runs
       ↓
DOM updates
```

so the effect and UI never observe stale memo values.

---

## Derived Functions vs Memo Timing

Derived functions:

```js
const double = () => count() * 2;
```

* Evaluated immediately when called
* No caching
* No batching

Memoized values:

```js
const double = memo(() => count() * 2);
```

* Cached between reads
* Dependency updates are batched
* Recomputed asynchronously
* Updated before effects and DOM rendering

This behavior allows Core to reduce unnecessary recomputation while keeping derived state consistent throughout the application.
