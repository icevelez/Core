# Template Syntax 

Core uses an HTML-based template syntax inspired by Handlebars that allows you to declaratively describe how the DOM should be rendered based on reactive state.

Templates in Core are designed to remain close to standard HTML, making them easy to read, write, and integrate with existing tooling and server-rendered environments. Under the hood, Core compiles these templates into optimized render function that work together with its fine-grained reactivity system to update only the parts of the DOM that depend on changed state.

In addition to the default Handlebars-style syntax, Core’s template layer is designed to be flexible. It can support alternative directive styles such as:

- Attribute-based directives (`x-if`, `x-for`) inspired by Vue and Alpine
- Annotation-style directives (`@if`, `@for`) inspired by Blazor and Angular

These syntaxes are interchangeable at the template layer and can be chosen based on developer preference.

However, for the purpose of this guide, we will focus on the default Handlebars-style syntax.

## Text Interpolation

The most basic form of data binding is text interpolation using the Handlebar syntax (double curly braces):

```html
<span>Greetings! {{ message }}</span>
```

The handlebar tag will be replaced with the value of the `message` variable from the corresponding component instance. It will also be updated whenever the message variable changes.

## Raw HTML

By default, interpolation expressions are rendered as plain text rather than HTML. To render a string as actual HTML, we use the `html` element directive:

```html
<p>Using text interpolation: {{ raw_html }}</p>
<p>Using the "html" action: <span use:html="raw_html"></span></p>
```

This introduces a new concept called an **Action**.

Actions are special attributes prefixed with `use:` that attach custom behavior to DOM elements. In this example, the `html` action keeps the element's `innerHTML` synchronized with the value of `raw_html`.

> [!WARNING]
> Rendering arbitrary HTML can introduce [**Cross-Site Scripting (XSS)**](https://owasp.org/www-community/attacks/xss/) vulnerabilities. Only use `use:html` with trusted content, and never with untrusted user input.

## Attribute Bindings

The `:` prefix tells Core to evaluate an attribute value as a JavaScript expression instead of treating it as a plain string.

```html
<div :value="dynamic_value"></div>
```

Attribute bindings are useful for dynamically setting element attributes, classes, styles, and other properties.

For example, conditional classes can be written as:

```html
<div :class="is_large() ? 'text-lg font-bold' : 'text-base'"></div>
```

```html
<div :class="`border border-red-500 ${ is_large() ? 'text-lg font-bold' : 'text-base' }`"></div>
```

Binding expressions are evaluated within the scope of the current component instance.

## Boolean Attributes

Attribute bindings can also be used with boolean HTML attributes such as `disabled`, `checked`, `selected`, `readonly`, `required`, and many others.

```html
<input type="text" :disabled="is_loading()" />
```

When the expression evaluates to a truthy value, Core adds the attribute:

```html
<input disabled />
```

When the expression evaluates to a falsy value, Core removes the attribute:

```html
<input />
```

This makes boolean attributes easy to control using reactive state.

### Disabled Elements

```html
<button :disabled="is_saving()">
    Save Changes
</button>
```

### Checked Checkboxes

```html
<input
    type="checkbox"
    :checked="is_completed()"
/>
```

### Selected Options

```html
<option
    value="admin"
    :selected="role() === 'admin'"
>
    Administrator
</option>
```

### Required Inputs

```html
<input
    type="email"
    :required="requires_email()"
/>
```

### Readonly Inputs

```html
<input
    :readonly="is_locked()"
/>
```

## Dynamic Attribute Values

Attribute bindings are not limited to boolean attributes. They can be used with any standard HTML attribute.

### Image Sources

```html
<img :src="user_avatar()" />
```

### Links

```html
<a :href="profile_url()">
    View Profile
</a>
```

### Input Values

```html
<input :value="username()" />
```

### ARIA Attributes

```html
<button
    :aria-label="button_label()"
>
    Click Me
</button>
```

### Data Attributes

```html
<div
    :data-user-id="user_id()"
></div>
```

## What Can Be Bound?

In general, if an attribute can be written in HTML, it can be bound dynamically using the `:` prefix.

```html
<div
    :id="id()"
    :title="title()"
    :class="classes()"
    :style="styles()"
    :hidden="is_hidden()"
></div>
```

Core evaluates the expression and updates the attribute whenever its reactive dependencies change.

---

### Expressions Only

Binding expressions must consist of a single JavaScript expression.

An expression is any piece of code that evaluates to a value and could appear after a `return` statement.

The following examples are **not valid** because they contain statements:

```html
<!-- Variable declarations are statements -->
{{ var number = 1 }}

<!-- Control flow statements are not allowed -->
{{ if (condition) { return message; } }}
```

Instead, use expressions such as ternary operators:

```html
{{ condition ? message : '' }}
```

### Calling Functions

Functions exposed by the component can be called directly within interpolation and binding expressions.

```html
<span>
    {{ format_date(date) }}
</span>

<input type="date" :value="date_to_string(date)" />
```

## Element Directives

Element Directives are functions that are invoked when an element is created and inserted to the DOM. They allow you to attach custom behavior, integrate third-party libraries, register event listeners, or perform direct DOM manipulation.

Element Directives are applied using the `use:` prefix:

```html
<script>
    export default function() {
        const my_custom_behavior = (node) => {
            // element initialization

            return () => {
                // cleanup logic
            };
        };
    }
</script>

<div use:my_custom_behavior>...</div>
```

Element Directives are executed only once when the element is created. Changing the argument value later will not cause the action to run again.

> [!NOTE]
> Because HTML attribute names are case-insensitive, action names cannot not use PascalCase or camelCase. It is recommended to use `snake_case` instead.
>
> **Recommended:** `my_action`
>
> **Avoid:** `myAction`, `MyAction`
