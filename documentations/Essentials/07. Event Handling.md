# Event Handling

Core provides a simple and consistent way to handle DOM events using the `on:` prefix.

This syntax allows you to attach event listeners directly within your templates, keeping behavior close to the element it affects.

---

## Basic Usage

Event handlers are defined using the `on:` prefix followed by the event name.

```html
<button on:click="() => count.set(count() + 1)">
    Click Me
</button>
```

This is equivalent to attaching a standard DOM event listener, but written in a declarative form inside the template.

---

## Accessing Component Functions

Event handlers run in the scope of the current component, allowing direct access to state and functions:

```html
<script>
    export default function () {
        const [count, setCount] = signal(0);

        const increment = () => {
            setCount(count() + 1);
        };
    }
</script>

<button on:click="increment">
    Increment
</button>
```

---

## Event Object

The native event object is automatically passed to event handlers:

```html
<button on:click="(event) => console.log(event)">
    Log Event
</button>
```

You can use it like standard DOM events:

```html
<input on:input="(e) => setValue(e.target.value)" />
```

---

## Multiple Event Listeners

Each event is handled independently:

```html
<button on:click="handleClick" on:mouseenter="onHover">
    Button
</button>
```

---

## No Event Modifiers

Core does not support event modifiers (such as `.prevent`, `.stop`, `.once`, etc).

Instead, all event logic is handled explicitly inside the event handler.

### Example

Instead of:

```html
<!-- Not supported -->
<form on:submit.prevent="submitForm">
```

You should write:

```html
<form on:submit="(e) => { e.preventDefault(); submitForm(); }">
```

--- 

## Why No Modifiers?

Modifiers introduce implicit behavior that is separated from the actual event handler.

Core avoids this abstraction in favor of explicit control, where event behavior is fully visible in one place and logic is not split between template syntax and handler code to keep execution flow predictable

--- 

## Event Delegation Model

Core uses a **delegated event system** internally.

Instead of attaching individual listeners to every DOM element, events are registered through a centralized delegation layer and resolved at runtime.

This means:

- Fewer actual DOM event listeners
- Lower memory overhead for large lists and dynamic UIs
- More consistent event handling across dynamically created elements

From a developer perspective, event usage remains the same:

```html
<li on:click="handleClick">Item</li>
```

But internally, Core resolves events through a shared delegation system attached at a higher level in the DOM tree.
