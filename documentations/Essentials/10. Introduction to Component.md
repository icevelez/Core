# Components

Components are one of the fundamental building blocks of a Core application.

A component encapsulates a portion of your user interface together with its state, behavior, and lifecycle. By breaking an application into smaller reusable pieces, you can build complex interfaces that remain easy to understand and maintain.

A typical Core application consists of many components arranged in a hierarchy:

```text
App
├── Header
├── Sidebar
├── UserList
│   └── UserCard
└── Footer
```

---

## Defining a Component

Components are typically written as `.html` files.

A component consists of two parts:

- A `<script>` block containing the component logic
- A template containing the rendered HTML

```html
<script>
    import { signal } from "core";

    export default function() {
        const [count, setCount] = signal(0);
        const increment = () => setCount(count() + 1);
    }
</script>

<button on:click="increment">
    Count is: {{ count() }}
</button>
```

The function exported from the `<script>` block is called when a component instance is created.

---

## Mounting a Component

Components must be mounted before they can be displayed.

```js
import { component, mount } from "core";

const App = await component("App.html");

mount(App, "#app");
```

```html
<div id="app"></div>
```

When mounted, Core:

1. Loads the component
2. Compiles the template
3. Creates a component instance
4. Renders the template into the target element

---

## Local State

Components can maintain their own state using signals.

```html
<script>
    import { signal } from "core";

    export default function() {
        const [name, setName] = signal("World");
    }
</script>

<input
    value="{{ name() }}"
    on:input="(event) => setName(event.target.value)"
>

<h1>Hello {{ name() }}</h1>
```

Whenever the signal changes, Core automatically updates the affected DOM nodes.

---

## Using Components

Components can be imported and used inside other components.

```html
<script>
    import { component } from "core/handlebar";
    
    export const UserCard = component("./UserCard.html");
</script>

<UserCard />
```

Components may be reused as many times as needed.

```html
<UserCard />
<UserCard />
<UserCard />
```

Each instance receives its own state and lifecycle.

---

## Component Hierarchy

Components naturally form parent-child relationships.

```
App
├── Navigation
├── Dashboard
│   ├── UserTable
│   └── Statistics
└── Footer
```

This allows large applications to be organized into smaller, focused pieces.

As a general guideline:

- Components should have a single responsibility
- Components should be reusable when possible
- Large components should be split into smaller components

---

## Component Lifecycle

Core provides lifecycle hooks that run at specific stages of a component's existence.

```html
<script>
    import { on_mount, on_destroy } from "core";

    export default function() {

        on_mount(() => {
            console.log("Mounted");

            // clean up function when a component is teardown 
            return () => console.log("Mount cleanup");
        });

        on_destroy(() => {
            console.log("Destroyed");
        });
    }
</script>
```

Lifecycle hooks are useful for:

- Fetching data
- Setting up subscriptions
- Starting timers
- Cleaning up resources

## Props

Props allow data to be passed from a parent component to a child component.

Parent:

```html
<UserCard name="John" />
```

Reactive prop 

```html
<UserCard :name="name" />
```

Child:

```html
<script>
    export default function(props) {
        console.log(props.name);
    }
</script>
```

Props provide a way for components to communicate while remaining independent.

---

## Slots

Slots allow a parent component to provide custom content to a child component using the `<CoreSlot/>` element

Child:

```html
<div class="card">
    <CoreSlot/>
</div>
```

Parent:

```html
<Card>
    <h1>Hello World</h1>
</Card>
```

The content passed between the component tags will be rendered inside the slot.

---

## Component Naming

Component files are typically named using PascalCase.

```text
UserCard.html
NavigationBar.html
TodoList.html
```

When used in templates, the same name is used:

```html
<UserCard />
<NavigationBar />
<TodoList />
```

This helps distinguish components from native HTML elements.

---

## Components and Reusability

The primary goal of components is reusability.

Instead of repeating markup:

```html
<div class="user-card">
    ...
</div>

<div class="user-card">
    ...
</div>

<div class="user-card">
    ...
</div>
```

You can create a single component:

```html
<UserCard />
```

and reuse it wherever needed.

This keeps templates smaller, easier to maintain, and easier to reason about as applications grow.
