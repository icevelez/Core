# Introduction to Signals

One of Core's most important features is its reactivity system. Reactivity allows Core to automatically track state dependencies and update the UI whenever that state changes.

## Creating Reactive State

Reactive state is created using the `signal()` function imported from the `core` module from the **Quick Start** guide 

```js
import { signals } from "core";

const [count, setCount] = signal(0);
```

A signal returns an array with two functions:

* `getter` — reads the current value
* `setter` — updates the current value

```js
import { signals } from "core";

const [count, setCount] = signal(0);

console.log(count()); // 0

setCount(1);

console.log(count()); // 1
```

Signals can store any JavaScript value, including primitive and supported non-primitive data types such as Array and Plain Objects.

---

## Primitive Signals

Signals work with primitive values such as:

* String
* Number
* Boolean
* BigInt
* Symbol
* Null
* Undefined

```js
const [name, setName] = signal("John");
const [age, setAge] = signal(25);
const [enabled, setEnabled] = signal(true);

setName("Jane");
setAge(26);
setEnabled(false);
```

When a signal value changes, Core automatically updates any bindings, or templates that depend on it.

---

## Object Signals

Signals can also contain objects.

```js
const [user, setUser] = signal({
    name: "John",
    age: 25
});
```

When accessing an object signal, Core returns a reactive proxy instead of the original object.

```js
const proxy = user();

console.log(proxy.name); // John
```

This allows Core to track property access and updates at a fine-grained level.

---

## Lazy Nested Proxies

Core automatically creates nested proxies when they are accessed.

```js
const [user] = signal({
    profile: {
        name: "John"
    }
});
```

```js
const profile = user().profile;
```

The `profile` proxy is not created when the signal is initialized. Instead, it is created the first time the property is accessed.

This technique is known as **lazy proxy creation** and helps reduce unnecessary work for large object graphs.

---

## Proxy Identity

Because Core wraps objects in reactive proxies, the value returned by a signal is not the original object.

```js
const object = {
    name: "John"
};

const [getUser] = signal(object);

const proxy = getUser();

console.log(proxy === object); // false
```

The proxy and original object are different objects.

Likewise, each signal instance creates its own reactive container.

```js
const object = {
    name: "John"
};

const [userA] = signal(object);
const [userB] = signal(object);

console.log(userA() === userB()); // false
```

Even though both signals reference the same original object, each signal creates its own dependency container and proxy hierarchy.

This allows signals to track dependencies independently.

---

## Mutating Objects

Object properties can be updated directly.

```js
const [user] = signal({
    name: "John"
});

user().name = "Jane";
```

Core automatically tracks and notifies any consumers that depend on `name`.

Nested properties work the same way.

```js
const [user] = signal({
    profile: {
        name: "John"
    }
});

user().profile.name = "Jane";
```

Only consumers that depend on `profile.name` will be updated.

---

## Arrays

Arrays are also reactive.

```js
const [todos] = signal([
    "Buy milk",
    "Walk dog"
]);
```

Array mutations automatically trigger updates.

```js
todos().push("Wash dishes");

todos().splice(0, 1);

todos().sort();
```

Core tracks these operations and updates dependencies as necessary.

---

## DOM Update Timing

State updates are applied immediately, but DOM updates are batched and processed asynchronously.

Core schedules DOM updates using the browser's **microtask queue**.

```js
const [count, setCount] = signal(0);

setCount(1);

// DOM may not be updated yet
```

This allows multiple state changes to be grouped together, reducing unnecessary DOM work.

---

## Waiting for DOM Updates

When you need to access the updated DOM after a state change, use `next_tick()`.

```js
const [count, setCount] = signal(0);

setCount(1);

await next_tick();

// DOM is now updated
```

This is particularly useful when:

* Measuring elements
* Reading layout information
* Managing focus
* Interacting with third-party DOM libraries

Example:

```js
setCount(count() + 1);

await next_tick();

console.log(element.textContent);
```

By waiting for the next update cycle, you can safely interact with the latest rendered DOM.

# Question and Answer 

## Why Is the Getter a Function?

You may have noticed that signals are accessed using a function call:

```js
const [count, setCount] = signal(0);

console.log(count());
```

instead of:

```js
console.log(count);
```

This is intentional.

Core uses the getter function to know when a piece of state is being used. This allows Core to automatically update the correct parts of your application when that state changes.

```html
<span>Count: {{ count() }}</span>
```

When `count()` is called, Core knows that `<span>` depends on `count`.

If signals exposed plain variables instead:

```js
const count = signal(0);

console.log(count);
```

there would be no opportunity for Core to intercept the read and establish a dependency relationship.

The function call acts as an explicit reactive read.

You can think of:

```js
count()
```

as:

```js
read_current_value_and_track_dependency()
```

---

## Why Are the Getter and Setter Separate?

Signals returns an array with two separate functions:

```js
const [count, setCount] = signal(0);
```

rather than a mutable object or variable.

This separation provides two benefits.

### 1. Explicit Reads and Writes

Reading state:

```js
count();
```

Writing state:

```js
setCount(1);
```

are distinct operations.

This makes it easy to determine where state is being modified when reading code.

```js
setCount(count() + 1);
```

is immediately recognizable as a state update.

---

### 2. Primitives and Objects Share the Same API

The same pattern works regardless of the value being stored:

```js
const [count, setCount] = signal(0);
const [user, setUser] = signal({
    name: "John"
});
```

Both signals expose:

```js
getter()
setter(value)
```

which creates a consistent mental model across the framework.

> [!TIP]
> If you're familiar with [**Solid.js**](https://www.solidjs.com/), Core's signals follow the same getter and setter pattern. The getter function is not merely a container for a value, it is the mechanism that enables dependency tracking and fine-grained updates.

--- 

## Why Some Built-in Types Are Not Reactive

Core's reactivity system is designed around plain JavaScript objects and arrays.

```js
signal({});
signal([]);
```

These data structures expose their properties through normal JavaScript property access, allowing Core to track reads and writes using Proxy traps.

However, many built-in JavaScript classes behave differently:

```js
Map
Set
WeakMap
WeakSet
Date
URL
URLSearchParams
```

These types often store their internal state behind specialized methods, internal slots, or private implementation details that cannot be reliably observed through ordinary property access.

For example:

```js
const map = new Map();

map.set("name", "John");
```

The value is modified through the `set()` method rather than a property assignment:

```js
map.name = "John";
```

Similarly:

```js
const date = new Date();

date.setHours(12);
```

changes the date's internal state without modifying any observable property.

Because these mutations occur through built-in methods and internal mechanisms, Core cannot reliably track them using the same dependency-tracking system used for objects and arrays.

This limitation is not unique to Core. Other reactive frameworks face similar challenges:

- Vue provides special handling for `Map` and `Set`
- Svelte provides reactive wrappers such as `SvelteMap`, `SvelteSet`, and `SvelteDate`
- Solid recommends storing plain objects and arrays in signals and stores

For consistency and predictable behavior, Core currently supports only:

```js
Primitive Values
Plain Objects
Arrays
```

If you need to work with unsupported types, consider converting them into reactive objects or arrays:

```js
const [users] = signal([
    ["john", { age: 25 }],
    ["jane", { age: 30 }]
]);
```

instead of:

```js
const [users] = signal(new Map());
```

Future versions of Core may introduce specialized reactive wrappers for selected built-in types, but plain objects and arrays remain the recommended foundation for application state.
