# Component Context

Context Hooks provide a way to share values between components without having to pass them through every level of the component tree using props.

A parent component can make data available to its descendants using `set_context()`, and any descendant can access that data using `get_context()`.

This is useful for values that are needed by many components, such as:

- Theme settings
- Authentication state
- Localization
- Router instances
- Shared services
- Form state

---

## Providing Context

Context is provided using the `set_context()` hook.

```js
import { set_context } from "core";

export default function() {
    set_context("theme", "dark");
}
```

The value becomes available to all descendant components.

---

## Consuming Context

Descendant components can access the nearest matching context value using `get_context()`.

```js
import { get_context } from "core";

export default function() {
    const theme = get_context("theme");

    console.log(theme); // "dark"
}
```

Context values are resolved by walking up the component hierarchy until a matching key is found.

---

## Example

### Parent Component

```html
<script>
    import { set_context } from "core";

    export default function() {
        set_context("user", {
            name: "John",
            role: "Administrator"
        });
    }
</script>

<UserProfile />
```

### Child Component

```html
<script>
    import { get_context } from "core";

    export default function() {
        const user = get_context("user");
    }
</script>

<h1>{{ user.name }}</h1>
<p>{{ user.role }}</p>
```

Output:

```text
John
Administrator
```

## Reactive Context

Context values can contain signals or other reactive objects.

### Provider

```js
import { signal, set_context } from "core";

export default function() {
    const [theme, setTheme] = signal("dark");

    set_context("theme", {
        theme,
        setTheme
    });
}
```

### Consumer

```js
import { get_context } from "core";

export default function() {
    const { theme, setTheme } = get_context("theme");

    const toggle = () => {
        setTheme(
            theme() === "dark"
                ? "light"
                : "dark"
        );
    };
}
```

Because the signal itself is shared, updates automatically propagate to all consumers.

---

## Context Shadowing

A descendant component may provide a new value for the same key.

```js
set_context("theme", "light");
```

Components below that provider will receive the new value instead of the ancestor's value.

```text
App (set value as "dark")
 └─ SideNav (receive value as "dark", set value as "light")
     └─ Button (receive value as "light")
```

In this example:

```js
get_context("theme");
```

inside `Button` returns:

```js
"light"
```

because it is the nearest matching provider.

---

## Context Keys

Keys may be strings:

```js
set_context("theme", value);
```

or symbols:

```js
const THEME = Symbol();

set_context(THEME, value);
```

Using symbols helps prevent accidental key collisions in larger applications and libraries.

---

## When to Use Context

Context is best suited for values that are shared across multiple levels of the component tree.

Good candidates include:

- Current user
- Theme configuration
- Router instance
- Localization settings
- Shared services
- Form groups

For parent-to-child communication where only a few components are involved, props are often the simpler choice.

---

## API Reference

### `set_context()`

Registers a context value for descendant components.

```js
set_context(key, value);
```

#### Parameters

| Name | Type | Description |
|--------|--------|--------|
| `key` | `string` or `symbol` | Context identifier |
| `value` | `any` | Value to provide |

---

### `get_context()`

Retrieves the nearest context value matching the provided key.

```js
const value = get_context(key);
```

#### Parameters

| Name | Type | Description |
|--------|--------|--------|
| `key` | `string` or `symbol` | Context identifier |

#### Returns

Returns the nearest matching context value.

If no matching provider exists, the return value is implementation-defined and may be `undefined`.

## Context Scope

Context values are only available to the component that provides them and its descendants.

```text
App
├── Header
├── Dashboard
│   ├── UserTable
│   └── UserCard
└── Footer
```

If `Dashboard` provides a context value:

```js
set_context("user", currentUser);
```

then:

```text
Dashboard
├── UserTable      ✅ can access
└── UserCard       ✅ can access
```

can retrieve it using:

```js
get_context("user");
```

However:

```text
Header  ❌ cannot access
Footer  ❌ cannot access
```

cannot access the value because they are outside the provider's component subtree.

---

### How Context Resolution Works

When `get_context()` is called, Core walks upward through the component hierarchy looking for the nearest matching context key.

```text
UserCard
    ↑
UserTable
    ↑
Dashboard
    ↑
App
```

The first matching provider wins.

For example:

```text
App
└── Dashboard
    └── UserCard
```

```js
// App
set_context("theme", "light");
```

```js
// Dashboard
set_context("theme", "dark");
```

```js
// UserCard
const theme = get_context("theme");
```

The value returned is:

```js
"dark"
```

because `Dashboard` is the closest provider.

---

### Context Is Not Global State

Context should not be viewed as a replacement for shared state.

Context is scoped to a particular branch of the component tree:

```text
Provider
├── Child A
│   └── Grandchild
└── Child B
```

All descendants of the provider can access the value.

Components outside that branch cannot.

```text
OtherComponent ❌
```

---

### Multiple Providers

Different branches can provide different values using the same key.

```text
App
├── AdminSection
│   └── set_context("theme", "dark")
│
└── PublicSection
    └── set_context("theme", "light")
```

Components inside each branch receive the nearest value.

```text
Admin Components  → dark
Public Components → light
```

This allows context to be localized to specific sections of an application.

> [!TIP]
> A useful mental model is that Context behaves similarly to variable scope in JavaScript. A component can access values provided by its ancestors, but not by its siblings or descendants.
