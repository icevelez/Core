## Introduction to Effects

Effects are the foundation of Core’s reactive execution model.

An effect is a function that automatically re-runs whenever any reactive dependency used inside it changes.

They are used to:

- Update the DOM
- Synchronize state
- Run side effects (logging, API calls, subscriptions)
- Coordinate derived reactive behavior

---

## Basic Usage

Effects are created using the `effect()` function:

```js
effect(() => {
    console.log(count());
});
```

Whenever `count()` changes, the effect will re-run automatically.

This works because Core tracks every signal accessed during execution and registers the effect as a dependent subscriber.

---

## Dependency Tracking

When an effect runs:

1. Core tracks all signals accessed inside the function
2. The effect is registered as a subscriber of those signals
3. When a signal updates, only dependent effects are re-executed

```
signal change
   ↓
notify dependencies
   ↓
queue affected effects
   ↓
flush microtask queue
```

---

## DOM Usage Example

Effects are commonly used for manual DOM interaction:

```js
effect(() => {
    titleElement.textContent = title();
});
```

When `title()` changes, only this effect runs again.

---

## Scheduling

All effects are scheduled asynchronously using the **microtask queue**.

This ensures:

- Multiple signal updates are batched
- DOM updates are optimized
- State changes remain predictable

```js
setCount(1);
setCount(2);
// effects run once in a batched update
```

---

## Effect API

```js
effect(fn, options);
```

### Options

```ts
options.track_inner_effect?: boolean
```

By default, effects created inside another effect are tracked and tied to the parent lifecycle.

When `track_inner_effect` is set to `false`, inner effects are **not automatically disposed by the parent effect**.

This gives full manual control over lifecycle management.

### Use Cases

- Conditional rendering (`if` blocks)
- List rendering (`each` blocks)
- Async rendering (`await` blocks)
- Custom lifecycle control

### Example

```js
let dispose;

effect(() => {
    if (dispose) dispose();
    
    if (visible()) {
        dispose = effect(() => {
            console.log("inner effect");
        });
    }
}, { 
    track_inner_effect: false 
});
```

In this mode, the inner effect must be manually disposed if needed.

> [!NOTE]
> This option is powerful but low-level. It is primarily used by Core’s internal rendering system and is not recommended for general application logic.

---

```ts
options.is_priority?: boolean
```

When enabled, this effect is executed with higher priority during the microtask flush cycle.

Priority effects run earlier in the queue to ensure that dependent derived computations such as `memo()` values remain in sync before other effects execute.

### Use Cases

- Ensuring computed/memo values are updated first
- Critical UI synchronization steps
- Internal framework coordination

### Example

```js
effect(() => {
    syncDerivedState();
}, {
    is_priority: true
});
```

---

## Execution Order

When signals update:

1. Priority effects
2. Normal effects
3. DOM commit phase

---

## Disposal

Every effect returns a cleanup function:

```js
const clean_up_effect = effect(() => {
    console.log(count());

    return () => {
        console.log("cleanup");
    };
});

clean_up_effect() // Stops effect from re executing
```

This cleanup runs when:

- The effect is re-executed
- The parent scope is disposed
- The effect is manually stopped

---

## Key Concepts

- Effects automatically track dependencies
- They re-run when reactive values change
- Execution is batched using microtasks
- Cleanup functions ensure no memory leaks
- Advanced options control lifecycle and execution priority
