# Components

Components are one of the fundamental building blocks of a Core application.

A component encapsulates a portion of your user interface together with its state, behavior, and lifecycle. By breaking an application into smaller reusable pieces, you can build complex interfaces that remain easy to understand and maintain.

A typical Core application consists of many components arranged in a hierarchy:

```text
App
├── Header
├── Sidebar
├── UserList
│   └── UserCard
└── Footer
```

---

## Defining a Component

Components are typically written as `.html` files.

A component consists of three parts:

- An optional `<script>` block containing the component logic
- A template containing the rendered HTML
- An optional `<style scoped>` block containing the component styling
 
```html
<script>
    import { signal } from "core";

    export default function() {
        const [count, setCount] = signal(0);
        const increment = () => setCount(count() + 1);
    }
</script>

<button on:click="increment">
    Count is: {{ count() }}
</button>

<style scoped>
    button {
        color: red;
    }
</style>
```

The function exported from the `<script>` block is called when a component instance is created. It is required if a `<script>` block is present

---

## Local State

Components can maintain their own state using signals.

```html
<script>
    import { signal } from "core";

    export default function() {
        const [name, setName] = signal("World");
    }
</script>

<input
    value="{{ name() }}"
    on:input="(event) => setName(event.target.value)"
>

<h1>Hello {{ name() }}</h1>
```

Whenever the signal changes, Core automatically updates the affected DOM nodes.

## Shared State

While components commonly manage their own local state, they can also consume state created outside the component.

This allows multiple components to share the same reactive data source.

```js
// state.js
import { signal } from "core";
export const [count, setCount] = signal(0);
```

Any component can import and use this state:

```html
<script>
    import { count, setCount } from "./state.js";

    export default function() {}
</script>

<button on:click="() => setCount(count() + 1)">
    Count: {{ count() }}
</button>
```

Because all components reference the same signal:

```text
Shared Signal
      ↓
 ┌────┼────┐
 ↓    ↓    ↓
Comp A Comp B Comp C
```

updates made in one component are immediately visible in every component using that signal.

---

### Example

Counter A:

```html
<script>
    import { count, setCount } from "./state.js";

    export default function() {}
</script>

<button on:click="() => setCount(count() + 1)">
    Increment
</button>
```

Counter B:

```html
<script>
    import { count } from "./state.js";

    export default function() {}
</script>

<p>Current Count: {{ count() }}</p>
```

When the button in **Counter A** is clicked, **Counter B** automatically updates.

---

### Local State vs Shared State

Local state belongs to a single component instance:

```js
const [count, setCount] = signal(0);
```

Each component receives its own independent state.

```text
UserCard A → count = 0
UserCard B → count = 0
UserCard C → count = 0
```

Shared state exists outside the component:

```js
export const [count, setCount] = signal(0);
```

All components reference the same signal.

```text
Shared count = 0
       ↓
All Components
```

Use local state when data belongs to a specific component.

Use shared state when multiple components need access to the same data.

> [!TIP]
> For simple applications, exporting signals from a module is often sufficient for state sharing. For larger applications, consider using Context or Manager for more structured state management.

## Using Components

Components can be imported and used inside other components.

```html
<script>
    import { component } from "core/handlebar";
    
    export const UserCard = component("./UserCard.html");
</script>

<UserCard />
```

Components may be reused as many times as needed.

```html
<UserCard />
<UserCard />
<UserCard />
```

Each instance receives its own state and lifecycle.

---

## Component Hierarchy

Components naturally form parent-child relationships.

```
App
├── Navigation
├── Dashboard
│   ├── UserTable
│   └── Statistics
└── Footer
```

This allows large applications to be organized into smaller, focused pieces.

As a general guideline:

- Components should have a single responsibility
- Components should be reusable when possible
- Large components should be split into smaller components

---

## Slots

Slots allow a parent component to provide custom content to a child component using the `<CoreSlot/>` element

Child:

```html
<div class="card">
    <CoreSlot/>
</div>
```

Parent:

```html
<Card>
    <h1>Hello World</h1>
</Card>
```

The content passed between the component tags will be rendered inside the slot.

---

## Component Naming

Component files are typically named using PascalCase.

```text
UserCard.html
NavigationBar.html
TodoList.html
```

When used in templates, the same name is used:

```html
<UserCard />
<NavigationBar />
<TodoList />
```

This helps distinguish components from native HTML elements.

---

## Components and Reusability

The primary goal of components is reusability.

Instead of repeating markup:

```html
<div class="user-card">
    ...
</div>

<div class="user-card">
    ...
</div>

<div class="user-card">
    ...
</div>
```

You can create a single component:

```html
<UserCard />
```

and reuse it wherever needed.

This keeps templates smaller, easier to maintain, and easier to reason about as applications grow.
