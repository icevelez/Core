# Composable Behaviors

Modern applications often need direct access to DOM elements to implement features such as focus management, drag-and-drop, intersection observers, animations, tooltips, and third-party library integrations.

Core provides **Element Directives** as a way to attach reusable and composable behavior to DOM elements.

An Element Directive is a function that receives a DOM element when it is mounted and can optionally respond to updates or perform cleanup when the element is removed.

Because directives are ordinary JavaScript functions, they can be shared across components, composed together, and reused throughout an application.

```html
<input use:focus />
```

```js
export function focus(element) {
    element.focus();
}
```

Element Directives are particularly useful when a behavior needs direct access to a DOM element but does not belong inside the component's state or template logic.

Common use cases include:

- Automatic focus management
- Tooltips and popovers
- Drag-and-drop interactions
- Intersection and resize observers
- Keyboard shortcuts
- Third-party library integrations
- Custom DOM behavior

Unlike event handlers, which respond to specific user interactions, Element Directives allow behavior to be attached directly to an element's lifecycle, making them a powerful primitive for building reusable UI functionality.

In the following sections, we'll explore how to create, configure, and compose element directives to build reusable behaviors across your application.

---

## Element Directive Arguments

An element directive can receive a value by assigning an expression to the directive.

```html
<input use:focus="true" />
```

```js
function focus(node, value) {
    if (value) {
        node.focus();
    }
}
```

The element directive signature is always:

```js
function directive(node, value) {

}
```

| Parameter | Description |
|------------|------------|
| `node` | The DOM element the directive is attached to |
| `value` | The value passed to the directive |

---

## Passing Multiple Values

If an element directive requires multiple arguments, pass an object.

```html
<input use:tooltip="{ text: 'Username', delay: 500 }" />
```

```js
function tooltip(node, options) {
    console.log(options.text);
    console.log(options.delay);
}
```

This is the recommended approach for configurable element directives.

---

## Example: Auto Focus

```html
<input use:auto_focus_on_mount />
```

```js
function auto_focus_on_mount(node) {
    node.focus();
}
```

When the component mounts, the input automatically receives focus.

---

## Example: Click Outside

```html
<div use:click_outside="close">
    ...
</div>
```

```js
function click_outside(node, callback) {
    document.addEventListener("click", (event) => {
        if (!node.contains(event.target)) {
            callback();
        }
    });
}
```

Element directives make it easy to package reusable DOM behavior into a single function.

---

## Example: Resize Observer

```html
<div use:resize="on_resize"></div>
```

```js
function resize(node, callback) {
    const observer = new ResizeObserver(() => {
        callback(node.getBoundingClientRect());
    });

    observer.observe(node);
}
```

Element directives are often used when working with browser APIs that require a DOM element reference.

---

## Element Directives vs Components

Element directives are intended for behavior.

```html
<input use:auto_focus />
```

Components are intended for structure and UI composition.

```html
<UserCard />
```

If you need to manipulate an existing DOM element, an element directive is usually the right choice. If you need to create new UI, consider using a component instead.

---

## When to Use Element Directives

Element directives are best suited for situations where direct access to a DOM element is required:

- Browser APIs
- Focus management
- Drag and drop
- Element measurements
- Custom event handling
- Third-party JavaScript libraries
- Reusable DOM behaviors

For most UI logic, reactive state and components are sufficient. Element directive should be reserved for behavior that specifically needs access to the underlying DOM node.
